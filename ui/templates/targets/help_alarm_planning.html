Alarm Planning (Target + Metrics)

How to design alarm scenarios before writing target SQL and metric rules.
Back to Help

Risk Posture First

Treat missed alarms as a higher-risk failure than extra alarms, but do not accept unlimited noise. A practical operating stance is: bias toward detection, then tune false positives down in controlled iterations.

  • Missed alarm cost: outage extension, compliance impact, customer impact, and delayed response.
  • False positive cost: operator fatigue, ticket churn, and reduced trust in alarms.
  • Goal: high recall with acceptable precision, measured continuously.

Target Design Checklist

  1. Define service objective: what business/system condition this target protects.
  2. Define blast radius: what fails if this condition is not detected.
  3. Select datasource reliability: verify data freshness and query stability.
  4. Pick SQL mode:
    Use single when one row represents one health state.
    Use multiline when one query returns many metric instances (host, queue, region, etc.).
  5. Set scheduler/timeout so runtime stays below interval with buffer.
  6. Define mute policy only for approved maintenance windows and explicit ownership.

Metrics Strategy: Alarming vs Monitor-Only

Not every metric should generate alarms. ANBU targets can include metrics that are collected for dashboards/trend analysis (for example Grafana) without triggering operational paging.

Metric Type Intent Configuration Guidance
Alarming metric Immediate operator response is needed. Define clear CRITICAL rules, actionable message, and correct action mapping.
Monitor-only metric Visibility/trending, no immediate action. Keep metric in target for historical analysis; avoid paging-style action setup.
Diagnostic metric Used to explain another alarming metric. Collect and chart it; use in message context or runbook links rather than direct alarming.

Threshold and Condition Planning

  • Start with known failure boundaries (SLO/SLA, capacity limits, saturation points).
  • Use baseline historical data to avoid thresholds based on one incident day.
  • Prefer explicit, explainable conditions over opaque complex formulas at first release.
  • For multiline metrics, verify each metric instance is uniquely mapped and tested.
  • Write alert messages for actionability:
    what failed, where, current value, threshold, and next step/runbook link.

False Positive Control Without Losing Coverage

  1. Deploy with conservative thresholds that detect likely incidents.
  2. Track first-week outcomes: true positive, false positive, missed detection.
  3. Tune one variable at a time (threshold, schedule, mute window, query semantics).
  4. Do not silence a noisy metric blindly; convert to monitor-only until root cause is understood.
  5. Keep audit trail and compare before/after changes in the Audit tab.

Operational Governance

  • Every alarming metric must have an owner, escalation path, and runbook.
  • Mute windows require reason, scope, start/end, and rollback checkpoint.
  • Review stale muted targets regularly; long mute periods are high risk.
  • Use Test Actions to validate scripts and payloads before production rollout.
  • Use History and Logs pages in post-incident review to improve precision/recall.
