AI-Assisted Alarm Migration

Fast process to migrate existing alarms into ANBU target files.
Back to Help

Fast Workflow (Recommended)

  1. Open Targets and click New Target With AI.
  2. Copy the prompt and paste it into your LLM.
  3. Provide source alarm definition (legacy query, thresholds, actions, severity policy).
  4. Ask for output in three files: .env, .sql, optional .hql.
  5. Paste generated files back into ANBU modal, then open target edit page and review.
  6. Test with Explore/Test Actions before enabling in production.

What to Keep from Existing Alarms

  • Preserve target intent and business impact semantics, not only threshold numbers.
  • Preserve severity mapping (CRITICAL/MAJOR/MINOR) unless there is a deliberate policy change.
  • Preserve alert messages that operators already understand, then improve placeholders and context.
  • Preserve known maintenance/mute behavior, but revalidate long mute windows.

Monitor-Only Metrics in ANBU

ANBU metrics are not always paging metrics. You can include metrics for trend visibility (for example Grafana) without operational noise.

  • Use non-paging action routing for normal and severity transitions where possible.
  • Avoid deleting useful metrics just because they are noisy. Convert them to monitor-only while tuning.
  • Keep metric naming stable across migration to preserve dashboard continuity.

Quality Gate Before Enable

  1. Validate SQL shape against chosen mode (single row vs multiline metric rows).
  2. Check mapping aliases and metric VALUE references.
  3. Run query in Explore and inspect result distribution.
  4. Confirm action scripts exist and are correct in Test Actions.
  5. Review mute windows and default schedule.
  6. Enable target with close observation for first cycles, then tune.
