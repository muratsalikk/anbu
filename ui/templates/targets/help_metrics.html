Metric Object Model

Each metric definition contains these core fields:

  • NAME: metric identifier. In multiline mode, this is matched against SQL row metric name.
  • VALUE: source reference used as main numeric value (VALn or alias).
  • NORMAL_ACTION + NORMAL_MSG: default path when no severity condition matches.
  • CRITICAL/MAJOR/MINOR condition arrays.

Usable Default Parameters in Formulas and Messages

Runtime builds a context before evaluating conditions and rendering messages.

Parameter Meaning
VAL1..VAL10Raw SQL columns.
Mapped aliasesFriendly names from SQL mapping tab.
TARGET_NAMECurrent target key.
METRIC_NAMECurrent metric being evaluated.
BASELINEMedian baseline from historical samples.
DEVIATIONMedian absolute deviation (MAD).
CONDITION_VALUEMatched threshold expression result.

Message templates support placeholders such as {{METRIC_NAME}}, {{BASELINE}}, {{DEVIATION}}, {{CONDITION_VALUE}}, and mapped aliases.

How Evaluation Works

  1. Resolve VALUE reference to metric numeric value.
  2. Calculate BASELINE/DEVIATION from historical results (same metric, day-of-week, and time window).
  3. Evaluate conditions in strict severity order: CRITICAL then MAJOR then MINOR.
  4. Inside each severity, evaluate condition rows by ascending index; first true condition wins.
  5. If no condition matches, severity becomes NORMAL and normal action/message are used.
  6. Severity number mapping: NORMAL=1, MINOR=2, MAJOR=3, CRITICAL=4.
  7. Final state may still be action_error, config_error, or data_error depending on action/data issues.

Expressions and Operators

  • Supported operators: =, <, >, =<, =>.
  • Condition value expressions support arithmetic and safe functions: max/min/greatest/least.
  • Expressions must evaluate to numeric values; unknown identifiers fail validation/runtime.
  • Legacy METRIC_n_SEVERITY_IF exists for backward compatibility, but structured condition arrays are preferred.

Single vs Multiline Metric Strategy

Single Mode

One SQL row is shared by all configured metrics. Use this when one snapshot row provides all values.

Multiline Mode

SQL returns one row per metric; runtime routes each row by first-column metric name.

To override behavior for a specific multiline metric, define a metric object whose NAME exactly matches that row's metric name (case-insensitive match).

For catch-all behavior, define a metric named DEFAULT. Runtime applies it to rows that do not match a specific metric definition. This is the recommended way to combine specific overrides with generic fallback logic.
