Engine Documentation

Implementation-level runtime behavior from scheduler tick to result row insertion.
Help Index Config Help

Source Files

  • eva/scheduler.py: scheduling loop, mute loop, per-minute dispatch.
  • eva/evaluator.py: target run pipeline and metric evaluation logic.
  • eva/engine.py: parsing helpers, condition expression evaluation, action execution.
  • eva/db.py: datasource connections, SQL execution, runtime row insertion.
  • eva/baseline.py: baseline/deviation calculation from historical runtime rows.

1) Scheduler Lifecycle

  1. Engine process starts with eva/eva.py and enters scheduler.main().
  2. Two loops run in parallel: scheduler loop dispatches evaluations each minute, mute loop precomputes dynamic mute state.
  3. Scheduler loads all target env headers from RULES_DIR.
  4. SCHEDULE_CRON is checked via croniter. Missing cron means run every minute.
  5. Each runnable target is submitted to run_target(...) in worker pool.

2) Mute Computation

Dynamic mute combines MUTE_BETWEEN_RULES and MUTE_UNTIL for the upcoming minute. Runtime then merges it with static IS_MUTED.

  • Between rules support multiple ranges and day filters.
  • Cross-midnight windows are supported.
  • Muted targets still evaluate and insert result rows, but actions are not executed.

Related tab docs: Config, History.

3) Target Run Pipeline (run_target)

  1. Load engine.properties, datasource definitions, and action definitions.
  2. Load target .env and validate required keys.
  3. Parse metrics and alias map from env.
  4. If inactive: insert skipped rows per metric and return.
  5. Read SQL file from QUERY_FILE (fallback to <TARGET>.sql).
  6. Validate read-only SQL and optional jitter delay.
  7. Execute SQL in selected datasource with timeout.
  8. Branch into single or multiline handling.

4) SQL Execution Contracts

Single Mode

  • Must return exactly one row.
  • Row columns map to VAL1..VAL10.
  • All configured metrics are evaluated against the same row.

Multiline Mode

  • Returns multiple rows, one metric row each.
  • Column 1 is runtime metric key; columns 2..10 map to VAL2..VAL10.
  • Duplicate metric keys in same run generate data_error with unexpected_multilne.
  • Rows can route to specific metric definitions by name, then fallback to DEFAULT metric definition if configured.

Related tab docs: SQL, Metrics.

5) Evaluation Context and Expression Safety

Context is built from SQL values, aliases, and runtime fields. Expressions are parsed through a restricted AST evaluator (safe arithmetic only).

  • Allowed functions: max, min, greatest, least.
  • No attribute access, list/dict access, lambdas, conditionals, or arbitrary function calls.
  • Unknown identifiers fail evaluation.
  • Operators are applied outside expression evaluation: =, <, >, =<, =>.

6) Severity and Condition Resolution

  • Severity order is fixed: CRITICAL then MAJOR then MINOR.
  • Within each severity, condition index order is ascending; first true condition wins.
  • If no condition matches, metric remains NORMAL.
  • Severity-to-number mapping: NORMAL=1, MINOR=2, MAJOR=3, CRITICAL=4.
  • Condition threshold result is exposed as CONDITION_VALUE for message rendering.

Related tab doc: Metrics.

7) Baseline and Deviation

For each metric evaluation, baseline stats are calculated from historical runtime rows:

  • Lookback window: last 3 months (config inside baseline function defaults).
  • Only same metric and same ISO weekday are considered.
  • Nearest minute-of-day samples are selected (up to max distance).
  • BASELINE is recency-weighted average (last month=3, prior month=2, third month=1).
  • DEVIATION is recency-weighted average absolute deviation using the same weights.
  • If baseline/deviation is required by formulas but unavailable, state can become data_error.

8) Action Execution

  • If muted: action is skipped and action name is stored as TARGET_MUTED-NO_ACTION_TAKEN.
  • If action name is not mapped in action properties: config_error.
  • If script returns non-zero or times out: action_error.
  • Action args: script, target_name, metric_name, metric_value, severity_number, rendered_message.

Related tab docs: Metrics, History.

9) Result Row States

Common state values written to runtime table:

  • ok: evaluation and action path succeeded.
  • nodata: SQL or value reference produced no usable data.
  • data_error: invalid runtime data pattern (for example duplicate multiline metric rows).
  • config_error: configuration/action mapping issues.
  • action_error: action script execution failed.
  • system_error: unexpected runtime exception.
  • skipped: target inactive.
