Read-Only Requirement

Target SQL must be read-only. Runtime accepts only SELECT or WITH ... SELECT patterns and rejects write/DDL statements.

  • Rejected examples: INSERT, UPDATE, DELETE, ALTER, DROP, CREATE.
  • Multiple statements separated by semicolons are rejected.
  • This validation is enforced in both UI validation and engine runtime checks.

Single Mode vs Multiline Mode

Mode Expected SQL Result Typical Use
single Exactly one row, up to 10 columns. Small fixed metric set evaluated from one snapshot row.
multiline Multiple rows. Column 1 must be metric name. Columns 2..10 are values. Per-metric row evaluation at scale (dynamic metric set, row-driven).

If your dataset naturally returns one row per metric (for example grouped KPI output), choose multiline. If your query returns one compact aggregate row consumed by all metrics, choose single.

Mappings: Why They Exist

Mappings convert positional SQL columns (VAL1..VAL10) into readable aliases (for example OPEN_COUNT, LAG_MIN). These aliases are then used in metric value binding, condition formulas, and message placeholders.

  • Without mappings, conditions use raw VAL* references.
  • With mappings, formulas are more maintainable and easier to review.
  • Aliases become part of evaluation context at runtime.

Multiline Mapping Rule

  • MAP_VAL1 is reserved and must be METRIC_NAME.
  • Do not assign METRIC_NAME to any other mapping slot.

Runtime in multiline mode reads column 1 as metric identity and starts value columns from VAL2. This is how per-row metric routing works.

Runtime Constraints and Errors

  • More than 10 returned columns results in config_error.
  • Single mode with 0 rows => nodata; multiple rows => data_error.
  • Multiline duplicate metric names in one run produce unexpected_multilne data errors.
  • Unknown datasource or invalid DSN produces config_error.
